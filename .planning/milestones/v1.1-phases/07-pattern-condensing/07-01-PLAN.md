**Merge family identification algorithm:**

The script must identify merge families by analyzing the 126 patterns. Use the following approach:

a) **Assessment family** (~12 patterns): Find patterns whose names contain "assessment" (case-insensitive). Group by topic area from enforcement_topics:
   - Security assessments (enforcement_topics includes data-security related topics)
   - Privacy assessments (enforcement_topics includes privacy related topics)
   - Cooperation with assessor (patterns about cooperation/working with third-party assessors)
   - Supplier assessment (if distinct, only 3 cases)
   Target: 3-4 merged groups from ~12 source patterns

b) **Acknowledgment family** (~10 patterns): Find patterns whose names contain "acknowledgment" or "receipt of order" or "distribution of order". All merge into 1 group: "Order Acknowledgments"
   Target: 1 merged group from ~10 source patterns

c) **Misrepresentation family** (~13 patterns): Find patterns whose names contain "misrepresentation" or "prohibition against". Group by topic area:
   - General misrepresentations (broad prohibition)
   - Privacy & security misrepresentations
   - Program-specific misrepresentations (about specific compliance program claims)
   - Security-only misrepresentations
   Target: 3-4 merged groups from ~13 source patterns

d) **Security/Privacy program family** (~12 patterns): Find patterns about security programs, privacy programs, comprehensive information security, etc. Group into:
   - Information Security Program
   - Privacy Program
   - Combined Security/Privacy Program (if distinct)
   Target: 2-3 merged groups

e) **Recordkeeping family** (~5 patterns): Patterns about recordkeeping, record creation, records maintenance. Merge into 1-2 groups.

f) **Compliance reporting family** (~7 patterns): Patterns about compliance reports, compliance reporting, notices. Merge into 1-2 groups.

g) **Corporate changes / distribution family** (~11 patterns): Patterns about corporate changes, distribution of order, successor notification. Group into:
   - Corporate Change Notification
   - Order Distribution
   Target: 2-3 groups

h) **Order duration / termination family** (~6 patterns): Patterns about order duration, sunset, termination, expiration. Merge into 1-2 groups.

i) **COPPA / Children family** (~8 patterns): Patterns about COPPA, children's data, KidzPrivacy. Group by:
   - COPPA Injunctions
   - Children's Data Deletion
   Target: 2-3 groups

j) **Monetary family** (~8 patterns): Patterns about civil penalties, monetary judgment, costs. Group into:
   - Civil Penalty
   - Monetary Relief / Judgment
   Target: 2-3 groups

k) **Deletion family** (~7 patterns): Patterns about data deletion, biometric deletion, information destruction. Group into:
   - Data Deletion Requirements
   - Biometric Data Deletion
   Target: 2-3 groups

**IMPORTANT implementation notes:**
- The script should read the actual pattern data and match by `id` and `name` fields, not by hardcoded assumptions. Use name-based regex/substring matching to find family members.
- For each merge group, compute `estimated_unique_cases` by collecting all unique `case_id` values across all variants of all source patterns (deduplication is critical -- see Pitfall 1 in RESEARCH.md).
- Compute `variant_count` as the sum of all source pattern variant_counts.
- Compute `year_range` from the min/max years across all source patterns.
- For the `target_id`, use slugify on the `target_name`.
- Patterns not matched by any family and not pruned remain untouched (passthrough).

**Prune candidate identification:**

Apply composite threshold: `case_count < 5 AND most_recent_year < 2020`. Research indicates this prunes ~19 patterns. Include both structural and substantive patterns that meet the criteria. Do NOT prune patterns that fail either condition (e.g., 4 cases but seen in 2024 should NOT be pruned).

**Console output:**

Print a clear, readable summary:
- Total patterns analyzed
- Merge families found (count, with names)
- Prune candidates found (count)
- Projected final pattern count
- A formatted table of merge groups with source counts and estimated case counts
- A formatted list of prune candidates with reason

The console output is what the user sees to decide whether to approve.
